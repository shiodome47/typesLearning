バケツリレーを無くす魔法のトンネル：
Context API

createContext は、コンポーネントツリーの裏側に「データの専用トンネル」を作ります。
親（Provider）がトンネルにデータを流し込み、子孫（useContext）は中間のコンポーネントを飛び越えて、どこからでも直接データを受け取ったり、更新関数を呼び出したりできます。

2. コードの構造とベストプラクティス

① Context と Provider の作成
// トンネルの設計図（型）
type ThemeContextType = {
  theme: "light" | "dark";
  toggleTheme: () => void;
};

// 魔法のトンネル本体を作成
const ThemeContext = createContext<ThemeContextType | null>(null);

// トンネルにデータを流し込む装置（Provider）
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState("light");
  const toggleTheme = () => { ... };

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children} // ← ここに入る全ての子孫がデータを使える
    </ThemeContext.Provider>
  );
}
② 安全な呼び出し（カスタムフック）
// トンネルからデータを取り出す「専用の蛇口」を作る
function useTheme() {
  const ctx = useContext(ThemeContext);
  
if (!ctx) throw new Error("ThemeProviderの外では使えません");
  return ctx;
}

// 実際のコンポーネントでの使用（Propsリレー不要！）
function ThemeToggleButton() {
  // 専用の蛇口をひねるだけでデータが出てくる
  const { theme, toggleTheme } = useTheme();

  return (
    <button onClick={toggleTheme}>
      現在のテーマ: {theme}
    </button>
  );
}

なぜわざわざ「useTheme()」を作るの？

コンポーネント側で毎回 useContext(ThemeContext) と書くこともできますが、ご提示いただいたコードのように カスタムフック（useTheme） を自作するのが現在のReact開発のベストプラクティス（定石）です。
万が一、開発者が <ThemeProvider> で囲み忘れた場所でこの機能を使おうとした場合、ctx が null になって原因不明のエラーでアプリが落ちてしまいます。カスタムフックの中で if (!ctx) throw new Error(...) とガードをかけておくことで、「囲み忘れてるよ！」とわかりやすいエラーを開発者に教えてくれる、極めて安全な設計になります。
