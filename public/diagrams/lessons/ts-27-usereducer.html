堅牢な状態管理パターン

命令を投げて状態を作る：
useReducer の図解

useState が「状態を直接書き換える」のに対し、useReducer は「やりたいこと（Action）をメモに書いて投げる（Dispatch）」だけです。受け取った工場（Reducer）が、安全な手順（switch文）に従って新しい状態を作ってくれます。

1. Dispatch と Reducer の動作シミュレーション

Action
<Counter /> (UI)
現在の State
{ count: 0 }
アクションを Dispatch (送信)
Dispatch
function reducer
引数1: state 引数2: action
switch (action.type) {
case "increment":
return { count: state.count + 1 };
case "decrement":
return { count: state.count - 1 };
case "reset":
return { count: action.payload };
}
新しい State を返却

2. 直和型（Discriminated Union）による圧倒的な安全性

Actionの型定義（直和型）
type Action =
  | { type: "increment" }
  | { type: "decrement" }
  | { type: "reset"; payload: number };

この書き方がTypeScriptにおける**直和型**です。
共通のプロパティ（ここでは type）を見ることで、TypeScriptは「今どのケースにいるか」を正確に判定します。

✅ 安全な推論（Type Narrowing）
switch(action.type) で "reset" に分岐した瞬間、TypeScriptは「このActionは絶対にpayloadを持っている！」と確信するため、安全に action.payload にアクセスできます。
❌ コンパイルエラーで守ってくれる
もし "increment" のケースの中で間違えて action.payload を使おうとすると、**「incrementアクションにはpayloadなんて存在しないよ！」**と赤い波線で怒ってくれます。実行する前にバグに気づけるのです。
なぜ useState ではなく useReducer なの？
カウンターくらいなら useState でも十分ですが、状態が「ユーザー情報」「カートの中身」「UIの開閉状態」など複雑に絡み合ってきた時、コンポーネント内に更新ロジックを散らかさず、「状態を更新するルール」を Reducer という純粋な関数に一極集中させることで、テストがしやすくバグの少ない設計になります。
