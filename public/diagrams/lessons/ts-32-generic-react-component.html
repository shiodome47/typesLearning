汎用 (Generic) コンポーネント

リスト表示など「データの形が違っても使い回したいUI」を作る時、function List<T> のようにジェネリクスを使います。
親から User[] を渡せば、描画関数 renderItem には自動的に User 型が伝播し、安全にプロパティを取り出せます。

親コンポーネント (Propsを渡す)
function List<T>
function List <T> extends { id: string | number }
({ items, renderItem }: {
  items: T[],
  renderItem: (item: T
) => ReactNode
}) {
Type 'string' does not satisfy the constraint '{ id: string | number; }'.
return (
  <ul>
    {items.map((item) => (
      <li key={item.id}>
        {renderItem(item)}
      </li>
    ))}
  </ul>
);
}

2つの重要ポイント

なぜ extends {id...} が必要なの？

Reactでリスト（li）を描画する際、要素を一意に識別するための key プロパティが必須になります。
もし <T> のまま（どんな型でもOK）にしてしまうと、TypeScriptは「渡された型が本当に id を持っているか分からないよ！」とエラーを出します。extends で制約をかけることで、コンポーネント内で安全に item.id にアクセスできるようになります。

「型推論の連鎖」が起こる

親コンポーネントで items={users} と配列を渡した瞬間に、<T> は User 型に決定されます。
すると、同じ T を使っている renderItem: (item: T) => ... の引数 item も自動的に User 型だと認識されるため、呼び出し側でわざわざ型を書かなくても、エディタの補完（u.nameなど）がバッチリ効くようになります。
